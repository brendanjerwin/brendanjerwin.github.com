How I Structure Tests

Feb 9th, 2010

This came up at work today, and it seems worth posting.

The way I’ve settled on organizing my tests/specs: first, I create general category folders for major architectural chunks of my application. In this example I have Domain and Persistance.

Inside those, if needed, I’ll add further folders to mimic the namespace of the code under test. In this case I have a fairly shallow domain (although it’ll get deaper with time and I’ll restructure the tests).

Finally, for each module I’m specifying, I’ll create a folder and for each method or scenario (if the the behavior is bigger than a single method) I’ll create a fixture. Inside each fixture I’ll list the various specifications.

This all results in a nice organization when running tests and makes it easy to add new specifications as needed.
