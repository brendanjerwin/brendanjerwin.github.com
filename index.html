Announcing - accessorize.js

May 26th, 2011

I’ve been working on a new open-source project recently. It’s a JavaScript library designed to convert plain-old JavaScript objects (POJOs? Isn’t that one taken already?) into powerful models, full of tasty syntax and functionality.

What does it do?

At its most basic, accessorize.js provides accessor functions over simple JavaScript objects. It creates “JQuery-style” accessor methods which act like a Getter if no parameters are passed in, and like a Setter if a parameter value is passed in:

obj.property() - Getter mode

obj.property("new value") - Setter mode

Even though JavaScript has standardized ‘real’ properties, they are not available in any usable form in IE8 and below. Additionally, I believe the syntactic capabilities I’m able to give you offer some pretty compelling reasons to stay with the JQuery-style accessors.

Observable

Accessorized accessors are observable. There is a .subscribe method hanging off each one that can be used to register a function to be called whenever the property value changes:

obj.property.subscribe(function(newValue) { alert("value changed!")});

Chain-able Setter Calls

Accessorized accessors can be set in a call chain, allowing more concise and readable initialization blocks:

obj.property("value")
   .otherProp("other value")
   .stillAnother("and so on")

Underscore Integration

Accessorized accessors are integrated with underscore.js as well. If the Underscore object (simply _) is passed into the accessor, then it triggers a special Getter mode in which the property value is returned already wrapped in the Underscore wrapper. This enables extremely convenient access to the powerful capabilities Underscore provides.

obj.arrayProperty(_).each(function(val){ //do something here });

Fancy Array Accessors

Accessorized array accessors are even more capable. In addition to the other features, including Underscore integration, array accessors also offer indexing variants of the standard accessor calls.

An array accessor can be invoked with the standard Getter mode behavior, but if a number is passed in as a single argument, the array accessor will instead return the value of the array at that index. Like this:

obj.arrayProperty(["hello","world","accessorize.js","is here!"]); //set the array
alert(obj.arrayProperty(2)); //returns "accessorize.js" to the user

The indexing Getter mode also works with Underscore as well:

obj.arrayProperty(1,_) //returns the Underscore wrapped value from that index

Finally, the standard Array methods are promoted to the accessor itself, allowing the accessor to be used as if it was an array in many cases. Methods that cause the contents of the array to change, like .sort will, of course, cause any subscribers to be notified of the change.

obj.arrayProperty.sort(); //sorts the array and notifies any subscribers of the changes

What’s next?

The next major features I’ll be baking in are live DOM binding and integration with Handlebars.js.

The live DOM binding will leverage the observable accessors to make updates to your model’s values automatically appear in your UI.

Handlebars integration will leverage the DOM bindings, allowing you to easily establish live bindings in your templates.

Where is it?

accessorizejs.com is the project’s homepage. You can also go directly to the github site.

It’s written in CoffeeScript, so you’ll have to get one of the compiled releases from the downloads section on GitHub. Or compile it yourself.

Wiki-Managed Help Command

Sep 19th, 2010

Nu just got (at least in the develop branch, it’ll be released soon) a new help command. It works by taking in a sequence of subjects and displaying an article about them. Something like this: nu help commands install or nu help about authors or even nu help commands install conflicts.

My favorite part about this system is that the content is actually maintained in the GitHub wiki for the project. The new GitHub wikis are now stored as a git repository of files, which opens up all sorts of interesting possibilities. If someone wants to update some existing documentation they simply need to edit the page in the wiki and the next release can include their modifications. If someone wants to add an entirely new subject to the documentation all they need to do is add a page and the next release will pick that up too.

I added the wiki as a submodule of the main project in a docs folder. Then I modified the nu.gemspec to include the docs folder in the gem. At runtime I simply join all the command-line arguments with a dash (-), stick .md on the end and go look in the gem folder for the file. There is a tiny bit of post-processing to remove hyper-link markup and, eventually, to dereference cross-reference ‘See:’ links. There is an issue describing the cross-reference functionality.

The end result of all this is that Nu gets a great documentation resource that is available at runtime, available online, and can be conveniently maintained by more than just the core team. Win, win, and more win.

Dependency Version Leveling

Sep 9th, 2010

There is an open issue for Nu right now referring to “Dependency Leveling”. This is the idea that Nu will find the highest versioned set of dependencies that can work together for a given lib folder. There has been a bit of discussion on it, but since I’m about to start writing code, I thought it would be helpful to describe my thoughts here.

Here is an example from the mailing list (slightly edited):

nu install fluentnhibernate
nu install castle.windsor

FNH is built against NH 2.1.2 and that in turn runs against castle.core 1.1.0.0
Windsor 2.5 then is built against castle.core 2.5 and windsor 2.1 is built against 1.2
So the only compatible version of Windsor and FNH is Windsor 2.0.0.0 + NH 2.1.2.4000

The Ruby Gems package contains enough information to solve this problem but Nu is going to have to recognize when these version conflicts occur and fix them automatically or offer a solution to the user.

In this case, Fluent NHibernate and NHibernate were installed first; Fluent NHibernate will be at the most current version available and NHibernate will be at the most current version available that fits within Fluent’s constraints.

When Castle Windsor is installed, Nu needs to detect that Castle Windsor and NHibernate have a dependency in common, and then recognize that there is a version conflict.

Dependency Version Overlap

The first attempt at resolving the conflict will be to see if a the version requirements for Castle.Core in the installed version of NHibernate and the requested version of Castle Windsor have any versions in common. If so, Nu will install that version of Castle.Core. Nu will not ask for permission to do this.

Newcomer Downgrade

That won’t work in this example though, so Nu will start attempting the same Dependency Version Overlap check on the previous version of Castle Windsor. It will continue checking older and older versions until it reaches a predefined hard-stop or finds a compatible package. In our example, Nu would find that Castle Windsor 2.0.0.0 was compatible and install it. Nu would perform this downgrade automatically, without asking the user for permission.

Incumbent Dependency Downgrade

If the conflicting package is itself a dependency of another package, like NHibernate in our example, the next approach that Nu can try is to see if, by downgrading the conflicting package, it can find a compatible set. In this case, Nu will attempt to downgrade NHibernate, within the dependency requirements of all packages which depend on it, until it finds a compatible set or runs out of available options. For each compatibility check Nu will execute the entire stack of rules, i.e. it will first look for Dependency Version Overlap, and then attempt to downgrade Castle Windsor using Newcomer Downgrade. If Nu is able to find a workable set here it will also do so automatically, since the downgrades were within the declared compatible versions of the installed package (Fluent NHibernate).

Root Package Downgrade

The final approach Nu can take is to try the same set of operations, but attempt downgrades of the packages at the root of the dependency graph. It would, in our example, attempt to downgrade Fluent NHibernate and then run through the entire set of compatibility checks. It would continue downgrading Fluent NHibernate until it found a compatible set or hit a hard-stop. If Nu were able to find a compatible set, it would display the versions to the user and ask if they wanted to rebuild their lib folder to match.

These rules get increasingly expensive to execute, so I think a good approach would be to implement just the first two rules and see how things go. If we run into real-world situations that look like we would like to solve then we can implement the other rules one at a time.

Help!

I would love some thoughts on these rules. Some other examples (real or cooked up) that illustrate the edge conditions are also appreciated. This set of rules is important if Nu is going to be valuable to the .NET community. Like Daniel said, without it Nu “is just a glorified wget.”

Nubular (Nu) Command - Specification

Sep 6th, 2010

UPDATE: Released in v0.2.5.

Nubular (Nu) will be getting a new command soon: specification.

The specification command is intended to provide detailed package information to integrating applications. (Most notably NuForVS.)

It can be used to get package information from a lib folder (default), the local machine’s gem cache (--cache), or from the configured gem servers (--remote).

When querying either the local cache or remote servers, you can specify a particular version with the --version switch, or Nu will simply return the specification for the highest, non-pre-release, version available. It’s not valid to specify a version when querying against a lib folder.

The specification command always returns JSON even if the --json switch is not set, so don’t be surprised if you can’t read the output.

Nubular (Nu) - Using Ruby Gems to Manage .Net Libraries

Aug 29th, 2010

I’ve recently started hacking on a pretty cool project. Nubular, or nu, is a front-end over Ruby Gems that is intended to maintain third-party references in a .Net project’s Lib folder.

It utilizes the dependency resolution system already built into Ruby Gems to resolve, download, and install all the pieces and parts needed when adding a third-party reference.

For example, executing nu install fluentnhibernate produces a Lib folder with these packages:

The following packages are installed:
=====================================
    castle.core-1.1.0.0
    castle.dynamicproxy2-2.1.0.0
    fluentnhibernate-1.1.0.685
    log4net-1.2.10.0
    nhibernate-2.1.2.4000
=====================================

(That output was generated with nu report.)

It’s pretty amazing how quickly and easily I can go from 0 to ready to roll with all my standard OSS infrastructure downloaded and waiting for me.

The project’s wiki is here and the mailing list (Google Groups) is here. A list of currently available packages is here.

There is also a Visual Studio Add-In being developed! It should make things wonderfully easy to manage. You can get it here.
